关于https的s和加密算法

Posted on 2021-09-12

HTTPS

（之前以为https中的s就是指ssl，面完试发现还有一种tls。。。）
首先SSL（Secure Socket Layer）是基于HTTPS下的一个协议加密层。HTTP的一个安全隐患是传输数据时使用的都是明文（即便数据在报体内，也可以被抓包工具获取）。为了解决这个问题推出了SSL，是HTTP之下TCP之上的一个协议层，TLS是SSL3.0的标准化实现（其实可以说近似于SSL3.1？）。事实上我们现在用的基本都是TLS，但是历史上习惯了SSL这个称呼。

TLS和SSL的主要区别：

  1. 版本号
    二者记录格式相同，但是版本号的值不同
  2. 报文明鉴码
    SSL和TLS的MAC算法以MAC算法计算的范围不同（什么是MAC算法：含密钥的HASH函数，消息的散列值由只有通信双方知道的秘钥来控制）。TLS使用HMAC算法，SSL使用了类似的算法，但HMAC中填充字节和密钥之间的是异或算法，而SSL中是连接运算。
  3. 伪随机函数
    TLS中使用PRF伪随机函数将密钥扩展为数据块
  4. 报警代码
    TLS支持几乎所有SSL的报警代码，并且补充定义了很多报警代码（解密失败、记录溢出、未知、拒绝访问等）
  5. 加密计算
    解密主密值时采用的方式不同
  6. 填充
    用户数据加密之前需要增加的填充字节，在SSL中，填充后的数据长度要达到密文块长度的最小整数倍，而TLS中可以是任意整数倍（防止基于对报文长度进行分析的攻击）

HTTP采用对称加密和非对称加密并用的混合加密机制。在交换密钥环节使用非对称加密，通信建立起来之后使用对称加密。但是非对称加密中的公开密钥无法证实该公开密钥就是真实可信的公开密钥本身，所以需要使用由数字证书认证机构CA和其相关机构办法的公开密钥证书。

参考《图解HTTP》了解一下HTTPS的SSL握手过程：

  1. 客户端发送Client Hello报文开始SSL通信，报文中包括客户端支持的SSL的版本、加密组件列表（加密算法、密钥长度等）
  2. 服务器可以进行SSL通信时，会以Server Hello报文作为应答，和客户端一样，报文中包含SSL版本以及加密组件（从客户端发来的组件中筛选出来的）
  3. 服务器发送Certificate报文，报文中包括公开密钥证书
  4. 服务端发送Server Hello Sone报文通知客户端，最初阶段的SSL握手协商部分结束
  5. SSL第一次握手结束后，客户端以Client Key Exchange报文作为回应，报文中包括已经被公开密钥加密的随机密码串
  6. 客户端继续发送Change Cipher Spec 报文，提示服务器之后的通信采用步骤五中的随机密码串进行加密
  7. 客户端发送Finished报文，包含连接至今全部报文的整体校验值，握手协商能否成功，以服务器能否正确的解密该报文为准
  8. 服务器同样发送Change Cipher Spec报文
  9. 服务器同样发送Finished报文
  10. 客户端与服务端的Finished报文交换完毕后，SSL连接建立完成。然后开始应用层的协议

加密算法（待详细了解。。。）

MD5

使用哈希函数，对一段信息产生信息摘要，以防止被篡改

SHA1

消息摘要算法哈希算法

HMAC算法

密钥相关的哈希运算消息认证码，利用哈希算法，以一个密钥和一个消息为输入，生成一个消息摘要作为输出

AES

对称块加密算法

DES

分组密码，以64位为分组对数据加密，密钥长度56位

RSA

对两个大素数的乘积进行因式分解是非常困难的，所以可以使用乘积作为加密密钥

axios取消请求

Posted on 2021-06-27

一般来说，当同一个请求重复被发起时，最新发起的请求被视为有效请求，之前的任一未返回的请求可被视为无效请求而取消。
axios可以通过使用cancel token来取消请求。（cancel token基于草案[https://github.com/tc39/proposal-cancelable-promises]，应该是已经取消了）

取消请求的两种方法：

  1. 可以使用CancelToken.source工厂来创建cancel token
  2. 传递executor函数到CancelToken的构造函数来创建cancel token

可以使用同一个cancel token来取消多个请求。

  1. axios对象如果做到既能做对象使用，又能做 函数使用

    axios.get(url[, config])和axios(config)

    在创建实例createInstance中

    instance其实是一个绑定了this的函数，调用axios其实是调用context.request -> 做函数使用

    这一行将Axios.prototype上的方法扩展到instance对象上，所以axios.get相当于Axios.prototype.get（绑定了context）

  2. 拦截器的主要逻辑

    • 发送请求的整个执行顺序是，requestInterceptors –> dispatchRequest –> responseInterceptors
    • 拦截器最初接收的对象是config，axios使用中也规定，请求的拦截器必须要返回config，这也是每个请求拦截器的函数参数是config的原因
    • 拦截器的执行顺序与interceptors.request.use(function () {/*...*/})执行的顺序有关,即先use的请求拦截器会先执行。
    • 如果拦截器中的函数时async函数，会阻塞整个拦截器链的执行，而transformData不会，所以如果需要对请求的数据做异步处理的话，要在拦截器中完成。

  3. dispatchRequest的核心逻辑

    • 处理config
    • 使用adapter发送请求,axios默认内置两个adapter,一个是负责在brower发送请求的，一个是负责在node端发送请求，可以在根文件的defaults下看到
    • 构造响应数据

    adapter是可以替换的（处理缓存的adapter）
